# План: агенты и правила для саундчека и концерта (WING OSC), без мешанья звукорежиссёру

Документ для руководителя проекта (без обязательного знания программирования).  
Техническая реализация — в коде (`backend/`, `config/`, `docs/`).

---

## 0. Главная идея (одной фразой)

**Компьютер предлагает и аккуратно помогает; финальное слово и руки на пульте — у человека.**  
Автоматика не «перехватывает» пульт, а работает в **режиме согласования** и **жёстких лимитов безопасности**.

---

## 1. Принципы (нельзя нарушать)

| Принцип | Простыми словами |
|--------|-------------------|
| Человек главный | Любое серьёзное изменение — только после явного «да» (или заранее выбранный доверенный режим на ограниченный набор действий). |
| Безопасность слуха | Нельзя бездумно крутить громкость вверх; есть потолки и проверки (в т.ч. true peak / лимиты в конфиге). |
| Обратная связь важнее микса | Если детектор обратной связи срабатывает — у него приоритет (как уже задумано в проекте). |
| Не мешать рукам на пульте | Команды OSC — редкие, пакетные, с задержкой между группами; не спамить пульт. |
| Прозрачность | Каждое действие агента: что, почему, какой канал, какое значение — видно в логе и в интерфейсе. |

---

## 2. Режимы работы с пультом (чтобы не мешать звукорежиссёру)

Рекомендуется три уровня (можно назвать в интерфейсе понятно):

1. **Только советы (Suggest)**  
   Агент анализирует, пишет список рекомендаций. **На пульт ничего не уходит.**  
   Инженер сам крутит или нажимает «применить выбранное».

2. **Мягкое применение (Assisted)**  
   Уходят **только** заранее разрешённые типы команд (например, мелкая подстройка одного параметра, или сцена из библиотеки с подтверждением).

3. **Ограниченный автопилот (только саундчек / репетиция)**  
   Короткие сценарии: «выставить стартовые гейны по шаблону», «прогнать тест каналов» — **только** после явного старта оператором и **тайм-аут / стоп-кран**.

На **концерте** по умолчанию держать **Suggest** или **Assisted**; полный автопилот — только если вы отдельно примете риски и протокол.

---

## 3. Архитектура «правила + агенты» (логика)

```
[Микрофоны / Dante] → [Анализ звука и состояния пульта]
                              ↓
                    [Правила + лимиты безопасности]
                              ↓
              [Координатор агентов — кто что предлагает]
                              ↓
         [Очередь действий: совет | запрос подтверждения | отправка OSC]
                              ↓
                    [WING по OSC]  ⟷  Звукорежиссёр вручную
```

- **Правила** — быстрые, предсказуемые «если А, то не делай Б» и «если С, предложи Д».  
- **Агенты** — более гибкая логика (в т.ч. с обучением и базой знаний), но **всегда** проходят через слой правил и режим.

## 4. Пошаговый план внедрения (фазы)

### Фаза A — База и безопасность (фундамент)

| Шаг | Что делаем | Зачем | Как понять, что готово |
|-----|------------|--------|-------------------------|
| A1 | Зафиксировать режимы Suggest / Assisted / (опц.) Auto для OSC | Чтобы по умолчанию не трогать пульт | В конфиге и в UI видны режимы |
| A2 | Каталог разрешённых OSC-адресов по функциям (фейдер, EQ, комп, мьют…) | Чтобы агент не «лез куда попало» | Таблица: функция → можно/нельзя в каждом режиме |
| A3 | Лимиты: макс. громкость, скорость изменений, пауза между пакетами команд | Не мешать инженеру и не ддосить пульт | Тест: ручное + автоматическое — пульт отзывчив |
| A4 | Журнал: кто предложил / кто применил / время | Разбор инцидентов и обучение | Можно открыть историю за вечер |

**Обратная связь руководителю:** короткий отчёт «режимы есть, лимиты включены, лог пишется».

---

### Фаза B — Правила по функциям пульта

Для **каждой** группы функций — отдельный блок правил (YAML или таблица + код):

| Функция | Примеры правил (идея) |
|---------|------------------------|
| Фейдеры | Не поднимать выше X дБ без запроса; при фидбеке — только снижение (уже в концепции проекта). |
| Панорама | Малые шаги; не трогать во время речи ведущего, если детектор «голос» активен (пример). |
| EQ | Ограничить глубину коррекции на канал; запрет экстремальных добавок на сабе в лайве. |
| Компрессор / гейт | Только из пресетов или в узких пределах ratio/threshold. |
| Мьют / соло | Соло — только в саундчеке; в концерте — только предупреждение в лог. |
| Сцены / снапшоты | Загрузка полного снапшота — только с явного подтверждения (опасная операция). |
| Маршрутизация | Только предложения; применение — вручную или отдельный «опасный» флаг. |

**Шаги:**

| Шаг | Действие |
|-----|----------|
| B1 | Составить список всех типов команд, которые проект уже умеет или планирует по WING. |
| B2 | Для каждого типа — колонки: Suggest / Assisted / Auto-саундчек / Концерт. |
| B3 | Внедрить проверку перед отправкой OSC: «команда разрешена в текущем режиме и сцене?» |
| B4 | Раз в спринт — ревью с реальным инженером: что мешало, что спасло. |

**Обратная связь:** таблица «функция × режим = ок/запрещено/под вопросом».

### Фаза C — Агенты под задачи саундчека и лайва

| Агент (роль) | Саундчек | Концерт |
|--------------|----------|---------|
| **Координатор** | Собирает цели («проверить мониторы», «линия барабанов») | Следит за приоритетами, не даёт конфликтующих команд |
| **Баланс / гейн** | Предлагает стартовые уровни по шаблону жанра | Мягкие подсказки при провале уровня |
| **EQ** | Рекомендации по ясности / низу | Осторожно, только если есть измерения |
| **Фидбек / безопасность** | Тест сигналов | Высший приоритет снижения |
| **Сцены** | Помощь с именованием / поиском снапшота по кэшу имён | Только советы, без автозагрузки снапшотов без «ОК» |

**Шаги:**

| Шаг | Действие |
|-----|----------|
| C1 | Описать сценарии саундчека списком (чек-лист). |
| C2 | Привязать каждый пункт к одному или нескольким агентам + правилам. |
| C3 | Все исходящие OSC — через единый «диспетчер» с rate limit. |
| C4 | UI: очередь предложений, галочки «применить», отмена. |

**Обратная связь:** демо-видео или чек-лист «сценарий саундчека пройден с подсказками».

### Фаза D — Обучение и база знаний (автоматизация цикла)

Идея: **не «агент сам учится на концерте без контроля»**, а **контролируемый конвейер материалов** в базу знаний (и при необходимости — в обучение моделей).

| Этап цикла | Что происходит | Кто решает «в прод» |
|------------|----------------|---------------------|
| **Поиск** | Скрипты/агент ищут PDF, статьи, разделы мануалов WING, ваши заметки | Автоматически собирается черновик |
| **Изучение / разбор** | Извлечение текста, разбиение на фрагменты, теги (EQ, OSC, сцены…) | Авто + выборка |
| **Проверка** | Вы или ответственный смотрите выжимку | Человек: «ок в базу» / «мусор» |
| **Коммит в репозиторий** | Только проверенные файлы в `docs/` или `backend/ai/knowledge/` | Через git — как обычный код |
| **Поиск нового** | По расписанию или по запросу — обновление списка источников | Отчёт «что нового найдено» |

**Важно:** сырьё с сомнительной лицензией **не коммитить** целиком — только ссылки, краткие выдержки своими словами, или материалы с явной лицензией.

**Шаги:**

| Шаг | Действие |
|-----|----------|
| D1 | Список доверенных источников (мануал WING, ваши PDF, внутренние регламенты). |
| D2 | Шаблон «карточки знания»: вопрос → короткий ответ → ссылка на раздел. |
| D3 | Автоматический отчёт раз в неделю: «добавлено / на проверке / отклонено». |
| D4 | Связка с RAG (у вас уже есть направление ChromaDB + правила) — только после D2. |

**Обратная связь:** еженедельное письмо/сообщение: 3–5 пунктов «что вошло в базу».

### Фаза F — Синхронизация с MacBook (чтобы ничего не терялось)

**Золотое правило:** единый источник правды — **GitHub + ваш локальный клон на Mac**.

| Шаг | Действие (простыми словами) |
|-----|------------------------------|
| F1 | На Mac в папке проекта один раз настроить `git pull` перед работой. |
| F2 | Включение Mac → открыть Cursor → **Подтянуть изменения** (pull) из нужной ветки. |
| F3 | Всё новое из облачных агентов попадает в репозиторий **только через коммиты** — после pull оно у вас в папке. |
| F4 | Локальные секреты и тяжёлые датасеты — **не** в git; переносить отдельно (флешка, облако), список в `README` или внутренней заметке. |
| F5 | (Опционально) Напоминание в календаре: «утро — git pull в проекте Tubeslave». |

**Если что-то не подтянулось** — почти всегда причина: не сделали pull, или работа была в другой ветке.

## 5. Как вы даёте фидбек по прогрессу (удобный формат)

Предлагается зафиксировать **3 уровня отчёта**:

1. **После каждой фазы (A–F)** — 10–15 строк: что сделано, что отложено, риски.  
2. **Еженедельно** (если идёт активная разработка) — статус по чек-листу: зелёный / жёлтый / красный.  
3. **По инциденту** (раз в пульт ушло не то) — один лист: время, команда, режим, кто подтвердил, вывод.

Вы как руководитель отвечаете коротко: **«принято» / «нужно изменить приоритет» / «стоп по пункту X»**.

## 6. «Скиллы» и доступы (чек-лист)

Под «скиллами» в Cursor имеются в виду **инструкции для ИИ** + **доступы**. Минимальный набор:

| Нужно | Зачем |
|-------|--------|
| Доступ к репозиторию GitHub | Чтобы агенты коммитили в согласованные ветки |
| Правила веток (main / feature) | Чтобы не ломать рабочую версию |
| Секреты не в чате | API-ключи — только в локальных/серверных конфигах |
| Документ этот (`PLAN_…`) | Общая карта; обновлять по мере прохождения фаз |

*(Конкретные файлы skills в Cursor создаются у вас в настройках; этот план — что в них описать.)*

## 7. Реалистичные ожидания по срокам и сложности (без календарных обещаний)

- **Фазы A + B + часть C** — относительно прозрачны: много инженерной дисциплины и тестов с пультом.  
- **Полная автоматизация цикла знаний (D)** — зависит от качества источников и вашего времени на проверку.  
- **«Не мешать звукорежиссёру»** на 100% технически не гарантируется никогда: всегда нужен **режим по умолчанию «только советы»** и культура использования.
